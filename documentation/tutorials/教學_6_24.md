## 1. 系統總覽與三角色架構

RocketCom 的核心特性是 **「同一份 firmware、用編譯期角色切出三個 binary」**。
這讓主航電、備援航電、地面站共用感測器驅動、EKF、協定與測試，維護單一真相來源。

角色定義集中於 `Main_Code/Core/Inc/board_config.h`（純常數巨集，不依賴 HAL／RTOS，
故純邏輯模組與 host 測試也能 include）：

```c
#define ROLE_PRIMARY 0   /* 主航電：完整飛控 + GPS/磁力計/LoRa 發送 + 板間鏈路 */
#define ROLE_BACKUP  1   /* 備援航電：只跑 MCU-layer 飛控（IMU/baro/highG→EKF→FSM→點火）*/
#define ROLE_GROUND  2   /* 地面站：只「接收」LoRa 下行 + 自身 GPS → SD/Flash/USB-CDC */

#ifndef BOARD_ROLE
#define BOARD_ROLE   ROLE_PRIMARY   /* MAKE_ROLE_LINE：build-backup/build-ground 改此行 */
#endif
```

### 1.1 功能閘（Feature Gates）

`main.c` 用 `#if FEATURE_*` 把對應驅動的 init／task 包起來，編譯期決定哪些子系統存在，
**關閉的功能完全不佔 code/RAM/CPU**：

| 功能巨集 | 主航電 | 備援 | 地面站 | 說明 |
|---|:---:|:---:|:---:|---|
| `FEATURE_GPS` | 是 | 否 | 是 | NEO-M9N GPS（USART6） |
| `FEATURE_MAG` | 是 | 否 | 否 | MMC5983MA 磁力計（軟體 I2C） |
| `FEATURE_LORA_TX` | 是 | 否 | 否 | 下行遙測「發送」 |
| `FEATURE_LORA_RX` | 否 | 否 | 是 | 下行遙測「接收」 |
| `FEATURE_FLIGHT` | 是 | 是 | 否 | 飛控管線（EKF/FSM/點火） |
| `FEATURE_LINK` | 是 | 是 | 否 | USART2 板間鏈路 |
| `FEATURE_USB_CDC` | 否 | 否 | 是 | USB 虛擬序列埠串流給 PC |

設計上要留意的一個工程取捨：**W25Q128 Flash 與 E80 920MHz LoRa 共用 SPI3**。
因此即使備援／地面站不發遙測，SPI3 周邊本身不能關（它們仍要寫 Flash）；備援是「不啟用
E80 驅動並按住其 RST 釋放匯流排」，地面站則「啟用 E80 但走 RX，讀 FIFO 與寫 Flash 以
SPI3 互斥鎖序列化」。這段邏輯詳見 `board_config.h` 的註解與 `w25qxx.c`。

### 1.2 一份程式碼、三個 binary 的建置

頂層 `Makefile` 提供：

```
make            # 主航電（ROLE_PRIMARY）
make build-backup   # 備援（ROLE_BACKUP），輸出 *_backup.elf
make build-ground   # 地面站（ROLE_GROUND），輸出 *_ground.elf
```

`build-backup` / `build-ground` 的手法很單純但要懂：先備份 `board_config.h`，用 `sed`
把標有 `MAKE_ROLE_LINE` 的那行改成對應角色，建置，**完成後一律還原** `board_config.h`，
再把產出的 `.elf` 另存為對應名稱。STM32CubeIDE 使用者則建議改用第二個 build
configuration（preprocessor 加 `BOARD_ROLE=ROLE_BACKUP`）。

## 2. 即時架構（FreeRTOS / 中斷 / 雙緩衝 / CCM RAM）

系統基於 STM32F407 + FreeRTOS。設計核心是**高頻採樣、即時估算、安全開傘、非同步儲存**。

### 2.1 任務與實測頻率（以源碼為準）

| 任務 | 頻率 | 優先級 | 職責 |
|---|---|---|---|
| `defaultTask` | **1 kHz**（`wake_tick += 1U`） | Normal | 主控迴圈：對齊高頻感測器、步進 FSM、處理 GPS/磁力計、餵狗、廣播板間鏈路 |
| `EKF_Task` | ~10 Hz 喚醒、內部 1 kHz 回放 | High（CCM RAM） | 每收到一批（~100 ms / 100 筆）就回放預測＋量測更新 |
| `LoRaTelemetry_Task` | **5 Hz**（`LORA_TELEM_PERIOD_MS=200`） | Low | 打包 79-byte 遙測，經 E22 / E80 發送 |
| `DiagnosticTask` | ~50 Hz（20 ms） | Low | 序列埠自檢、堆疊水位 0.2 Hz 印出 |

主迴圈內以「`tick % 週期 == 0`」分頻派工：

```c
if (tick % FSM_STEP_PERIOD_MS == 0) { ... }   /* FSM_STEP_PERIOD_MS = 10U → FSM 100 Hz */
if (tick % LINK_TX_PERIOD_MS == 0)  { ... }   /* LINK_TX_PERIOD_MS  = 50U → 板間鏈路 20 Hz */
```

所以一句話記住節律：**主迴圈 1 kHz、FSM 100 Hz、板間 20 Hz、下行遙測 5 Hz、EKF 批次回放**。

### 2.2 中斷與 NVIC 優先級

嚴禁在任務層級用阻塞輪詢讀感測器。改用硬體定時器中斷精準控時：

- **TIM3 (3.2 kHz)**：讀高 G 加速度計 ADXL375（SPI1）。
- **TIM6 (1.6 kHz)**：讀 BMI088 加速度計（SPI2）。
- **TIM7 (2.0 kHz)**：讀 BMI088 陀螺儀（SPI2）。
- **USART6（GPS）/ USART2（板間鏈路）**：DMA 循環 + 空閒中斷（IDLE），幀結束才觸發背景解析。
- BMP388 氣壓計則於**主迴圈 task context** 輪詢（~200 Hz），非高頻 ISR。

FreeRTOS 防護重點：`configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY = 5`。優先級數值
0～4 物理優先級最高、**絕不可呼叫任何 FreeRTOS API**；5～15 才能安全呼叫 `*FromISR`。
專案中採樣定時器 NVIC = 6（安全臨界區以下），USB 中斷 = 4（高於 syscall 限制，內部禁呼叫 RTOS）。

### 2.3 Ping-Pong 雙緩衝（避免資料撕裂）

高頻 ISR 在背景寫、`defaultTask` 在前景讀。若共用同一塊記憶體，會讀到「前兩軸新、第三軸舊」
的撕裂資料。解法是雙緩衝：ISR 寫 A、就緒後切到 B 並設旗標，任務只讀「已填滿且不再被改」
的那塊，全程無鎖。以 ADXL375 為例：

```c
static volatile float   g_adxl_batches[2][32][3]; /* 雙緩衝 */
static volatile uint8_t g_adxl_active_batch = 0;
/* ISR 滿 32 筆即： active = !active; cnt = 0; ready_idx = !active; new_ready = 1; */
```

### 2.4 CCM RAM 優化

F407 有一塊 64 KB **CCM RAM** 直連 Cortex-M4 D-bus、零等待，但 **DMA 無法存取**。
專案把運算密集且不需 DMA 的 **EKF 狀態（`EKF_x`/`EKF_P`/`EKF_q`）、EKF 任務 TCB 與堆疊**
以 `__attribute__((section(".ccmram")))` 放進 CCM，把常規 SRAM 留給需要 DMA 的 SD 卡與序列埠。

## 3. C/C++ 語法核心

### 3.1 指標
驅動常以指標傳遞硬體控制塊，避免複製、可直接改實體：
```c
HAL_StatusTypeDef ADXL375_Init(SPI_HandleTypeDef *hspi);
hspi->Instance->DR = val;   /* -> 為「成員存取指標運算子」 */
```

### 3.2 packed 結構（協定封包的根基）
預設編譯器會在成員間插入對齊填充。但遙測／板間封包必須緊湊、不容填充，故用
`__attribute__((packed))`。**本專案兩種封包（遙測、板間）都是 packed**：
```c
typedef struct __attribute__((packed)) {
    uint8_t  sync0; uint8_t sync1; uint8_t seq; uint8_t state;
    uint32_t tick_ms;   /* 緊接 state，無填充 */
} ExamplePacket_t;
```

### 3.3 位元運算
```c
int16_t raw = (int16_t)((rx[1] << 8) | rx[0]); /* 拼高低位元組 */
reg |=  (1 << 3);   /* 置位 */
reg &= ~(1 << 3);   /* 清位 */
```

## 4. 感測與狀態估算

### 4.1 感測器套件

| 感測器 | 介面 | 角色 | 備註 |
|---|---|---|---|
| BMI088（IMU 加速度＋陀螺） | SPI2 | 主姿態 | TIM6 1.6k / TIM7 2.0k |
| ADXL375（高 G ±200g） | SPI1 | 升空/頂點鑑別 | TIM3 3.2k；SPI1 與 BMP388 共線需互斥 |
| BMP388（氣壓高度） | SPI | 高度 + 開傘備援 | 主迴圈輪詢 ~200 Hz；發射台鎖定 T0 溫補 |
| MMC5983MA（磁力計） | **軟體 I2C** | 發射台航向鎖定 | 因硬體 I2C 易鎖死，2026-06-03 改 bitbang + 100 Hz polling |
| NEO-M9N（GPS） | USART6 | 水平位置 / 授時 | DMA-to-idle、NMEA-0183（GGA/RMC） |

### 4.2 軸向單一真相：`sensor_axis.h`

每顆感測器物理擺放方向不同，若各處臨時推正負號遲早出錯。專案把所有「感測器原生軸 → 機體
軸（X=右、Y=前、Z=上，右手系）」的對應集中在 `sensor_axis.h` 的 inline 函式，驅動層只輸出
原生軸、反向修正集中於此。`tests/test_sensor_axis.c` 以「旋轉矩陣行列式必為 +1」鎖定正確性。
**要改軸向，只動這個檔，並用測試驗證，不要手推。**

### 4.3 EKF 與健康降級

`ekf.c`（高優先級、CCM RAM）以四元數做姿態、氣壓做高度、GPS 做水平位置、磁力計在發射台做
航向鎖定。資料流：高頻 ISR 充填雙緩衝 → `defaultTask` 重構對齊成 1 kHz 同步包、滿批送
`xEKFQueue` → `EKF_Task` 回放預測＋量測更新 → 輸出姿態/高度供 FSM。

可靠度設計兩條：
- `ekf_guard.h`：回傳健康位，異常（四元數失常、發散等）即非 0。FSM 收到 EKF 不健康會**切換
  到 raw-baro 降級鏈**（純氣壓高度 + 加速度觸發，不靠 EKF 高度）。
- 垂直通道發散自救：偵測 DIVERGE 超過 3 s 自動重置。

> 工程備忘（尚未上板驗證）：`docs/設計_ADXL375_EKF融合與SPI1並發.md` 指出一個前置阻斷項——
> BMI088/ADXL375 樣本單位是 g，但 `EKF_Predict` 以 m/s² 扣重力積分，靜置由校準抵銷（故 bench
> 正常）但**動態被低估約 9.81×**。融合 ADXL 前須先把加速度統一為 SI（m/s²）。

## 5. 板間鏈路與不對稱備援

主航電與備援航電是**兩塊相同的 MCU 板**，各自獨立跑飛控；它們用 USART2 硬體全雙工互通狀態，
靠「不對稱冗餘」協調開傘：**主決策、備補位**。

### 5.1 實體層（USART2，硬體 UART）

兩板各跑 `LINK_BAUD = 38400` 的 USART2（PA2=TX/PA3=RX），於排線把 TX/RX 交叉一次
（A.PA2→B.PA3、A.PA3→B.PA2）即可對接。F407 USART 無硬體腳位對調，故用排線交叉取代
「韌體互換腳位」，換取硬體 UART 的可靠度（不走軟體 bit-bang）。

### 5.2 協定（`link_proto.h` / `link_proto.c`，純邏輯）

板間封包沿用遙測框架慣例，但**用不同 sync（0xC3,0x3C）**與下行遙測（0xA5,0x5A）區隔，避免
地面站把兩種封包混淆。**固定長度 26 bytes**：

```
sync0 sync1 board_id seq fsm_state flags  tick_ms  h_est_cm v_est_cms baro_alt_cm a_z_cg  crc16
  1     1      1      1      1       1       4         4         4          4        2       2   = 26 bytes
```

`flags` 直接重用 `telemetry.h` 的 `TELEM_FLAG_*`（單一真相來源），備板據 `DROGUE_FIRED` /
`MAIN_DEPLOYED` 判斷主板是否已開傘。收端 `LinkRx_Feed()` 是逐位元組狀態機，內建 sync 對齊與
CRC-16/CCITT-FALSE 校驗，框架破損會自動重新找 sync。

### 5.3 不對稱開傘仲裁：`BackupGate`（`link.h` / `link.c`，純邏輯）

這是備援設計的核心，刻意保持純函式以便 `tests/test_link.c` 鎖定行為：

- **對端狀態鎖存**：`LinkPeer_OnPacket()` 一旦收到主板的開傘旗標，就**鎖存**
  `drogue_latched` / `main_latched`（即使之後鏈路掉線仍維持，因共用點火頭已被主板點過）。
- **備板點火閘**：`BackupGate_Step()` 每飛控週期呼叫一次。語意是：

  ```
  若 對端已開（peer_latched） → 永久抑制本事件（主板已點，二極體 OR 同一點火頭，避免雙擊）
  否則 本板 FSM 想開 → 起算寬限期 BACKUP_GRACE_MS（300 ms）
        寬限內收到對端開傘 → 抑制
        寬限到期仍沒收到（主板漏點 / 失聯） → 備板自行點火一次（之後不重複）
  ```

換句話說：**正常情況以主板為準、備板閉嘴；只有主板真的沒動作或失聯，備板才在 300 ms 後補位。**
這同時解決了「共用一個點火頭不能被點兩次」與「主板失效時仍要有人開傘」兩個矛盾需求。

## 6. 雙鏈路二進制遙測協定

下行遙測同時走兩條 LoRa 鏈路做冗餘：

| | E22-400T30S | E80-900M2213S |
|---|---|---|
| 頻段 | 433 MHz | 920 MHz |
| 介面 | UART3 透傳 | SPI3（SX126x 命令式） |
| 驅動 | `lora_e22.c` | `lora_e80.c` |
| 現況 | 運作中 | **停用**（與 Flash 共用 SPI3，RF 參數待上板；見第 1.1 節故障隔離） |

### 6.1 封包格式（`telemetry.h`）

**固定 79 bytes、packed、結尾 CRC-16/CCITT-FALSE**（poly=0x1021, init=0xFFFF,
無反射, xorout=0x0000）。sync = `0xA5,0x5A`。所有浮點量乘倍率轉整數（避免 `%f` 吃堆疊/CPU）：

| 量 | 單位/倍率 |
|---|---|
| BMI088 加速度 | mg（g×1000） |
| BMI088 角速度 | dps |
| ADXL375 高 G | cg（0.01g，×100；±200g 用 cg 防溢位） |
| 磁場 | mGauss（×1000，機體軸） |
| 高度/速度 | cm / (cm/s)（m×100） |
| 四元數 | ×10000 |

封包含 EKF 高度/速度/四元數、氣壓、IMU/高G/磁/GPS、電池、CPU 佔用、以及兩組健康位
（`health_bits` = EKF、`sensor_bits` = 感測器）與 `flags`：

```c
#define TELEM_FLAG_DROGUE_FIRED  0x01  #define TELEM_FLAG_MAIN_DEPLOYED 0x02
#define TELEM_FLAG_SD_ACTIVE     0x04  #define TELEM_FLAG_GPS_STALE     0x08
#define TELEM_FLAG_EKF_UNHEALTHY 0x10  #define TELEM_FLAG_SENSOR_FAULT  0x20
#define TELEM_FLAG_FAILSAFE      0x40  #define TELEM_FLAG_HOTSTART      0x80
```

### 6.2 收端對齊（`telem_rx.h`）
地面端逐位元組掃 `0xA5,0x5A` 對齊封包邊界、以 `seq` 偵測丟包、以 `crc16` 校驗，CRC 失敗自動
滑窗重新對齊。**同一套邏輯在 C（韌體）與 Python（`GroundStation/telemetry_decoder.py`）各實作一份**，
契約由 `tests/test_telem_rx.c` 與 `tests/test_telemetry.c` 鎖定。

## 7. 地面站 ROLE_GROUND

地面站硬體與航電板相同，但只做**接收與記錄**，不參與飛控。入口 `ground_station.c`：
`main.c` 的 `defaultTask` 在 `IS_GROUND` 時呼叫 `GroundStation_Run()`（不返回）。

職責：
- **雙鏈路 RX**：E22 433（UART3 透傳，`GroundStation_OnUart3RxEvent` 推入位元組環形緩衝）
  與 E80 920（SPI3 讀 FIFO），各自餵進 `telem_rx` 解析。
- **自身 GPS**（USART6）：地面站也有 GPS，用於對齊時間/位置基準。
- **時間對齊** `gs_timesync.h`：把兩條鏈路與自身 GPS 的時間戳對齊（契約由 `tests/test_gs_timesync.c` 驗證）。
- **記錄** `gs_log.c`：SD（CSV）+ Flash（二進位 append）雙路；`tests/test_gs_log.c` 驗證格式。
- **USB-CDC**：`FEATURE_USB_CDC` 開啟，把收到的遙測 + 自身 GPS 串流給 PC。飛行版不啟用
  （`MX_USB_DEVICE_Init` 被守住，不列舉、不進 USB 中斷、不佔 CPU）。

## 8. 主機端純邏輯測試體系

所有「純邏輯」（不依賴 HAL/RTOS/硬體）都抽成可在電腦上編譯執行的模組，**無需上板即可驗證飛控
邏輯**。`tests/Makefile` 用系統 `cc`、`-I../Main_Code/Core/Inc`、`-std=c11`、`-lm`，
與韌體共用同一份 header/.c。執行：`make host-test`（= `make -C tests run`，任一失敗即非零中止）。

目前 12 個測試：

| 測試 | 驗證對象 |
|---|---|
| `test_sensor_axis` | 軸向對應為合法旋轉（行列式 +1） |
| `test_fsm` | 飛行狀態機：升空/頂點/開傘/落地情境 |

## 10. 如何修改代碼

### 10.1 新增感測器
1. CubeMX 設定 CS / I2C 腳位並生成代碼。
2. 參考 `adxl375.c` 寫初始化與物理值讀取。
3. 高頻採樣加入定時器中斷、寫入 Ping-Pong 寫入端；低頻則於主迴圈輪詢。
4. 在 `sensor_axis.h` 補上「原生軸 → 機體軸」對應，並加 `tests/` 測試。

### 10.2 改開傘參數
打開 `Main_Code/Core/Inc/fsm.h`，改 `LIFTOFF_ACCEL_THRESHOLD`、`TARGET_MAIN_ALTITUDE`
等巨集，必要時在 `fsm.c` 的 `FSM_Step()` 調整轉移規則。改完跑 `make host-test`（`test_fsm`）驗證。

### 10.3 切換角色重編
改 `board_config.h` 的 `BOARD_ROLE`，或直接 `make build-backup` / `make build-ground`。

> 注意 `docs/改進計劃.md` 的待辦：`FSM_FAILSAFE_APOGEE_MS` 等失效保護計時必須以 OpenRocket
> 模擬重推後才可上飛，不可沿用預設值。

## 11. 編譯與工具鏈

- **工具鏈**：必須用 **STM32CubeIDE 內建的 arm-none-eabi gcc**，**不可用 Homebrew 版**
  （頂層 `Makefile` 會自動從 CubeIDE plugins 路徑偵測 gcc 與 `STM32_Programmer_CLI`）。
- **常用指令**：`make`（編譯）、`make flash`（ST-Link SWD 燒錄）、`make monitor`
  （自動偵測序列埠 @460800，可用 `PORT=/dev/cu.xxx` 覆寫，Ctrl+A K 離開）、`make host-test`、`make clean`。
- **編譯四步**：預處理 → 編譯（.s）→ 彙編（.o）→ 連結（.elf/.bin）。
- **CubeIDE 路徑**：手動新增原始碼後，於 C/C++ General → Paths and Symbols 把 header 目錄加入
  Include Paths、`.c` 目錄加入 Source Location，否則找不到檔。
