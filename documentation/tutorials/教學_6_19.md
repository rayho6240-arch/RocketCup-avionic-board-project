## 1. 系統整體架構 (System Architecture)

航電系統的設計核心是**高頻採樣、即時估算、安全開傘與非同步儲存**。我們基於 STM32F407 晶片，並運行 FreeRTOS 即時作業系統。

```
[硬體定時器中斷 (TIM3, TIM6, TIM7)] ----(Ping-Pong 雙緩衝區)----> [defaultTask (1 kHz 主任務)]
                                                                        |
                                                           (非阻塞佇列 xEKFQueue)
                                                                        v
[DataLogging_Task (低優先級)] <----(雙快取/FatFs)---- [EKF_Task (高優先級，CCM RAM)]
```

### 1.1 FreeRTOS 任務排程與多任務設計
系統內劃分了多個不同優先級的任務，以保證關鍵控制迴圈的即時性：
* **`defaultTask` (1 kHz 週期，優先級 `osPriorityNormal`)**：主控制任務，運行於 1ms 週期。負責將定時中斷收集到的高頻感測器資料進行**降採樣濾波**與**時間對齊**，重構出 1000 Hz 的同步資料包。當積累滿 100 筆資料（即 100ms 份）時，將其包裝並透過訊息佇列（message queue）非阻塞地發送給 EKF 任務。此外，該任務也負責以 100 Hz (10ms) 頻率步進飛行狀態機（FSM）、處理 GPS 序列埠資料流、電池電壓 ADC 採樣與 LED 狀態指示。
* **`ekfTask` (10 Hz 週期，優先級 `osPriorityHigh`)**：高優先級 EKF 估算任務。平時處於阻塞狀態，一旦從 `xEKFQueue` 接收到一組 100ms 的資料包指標，便立即喚醒，在背景以 1000 Hz 的推進步長對該 100 筆資料進行回放（Playback），完成預測與量測更新，並輸出最新的姿態與高度估算。
* **`diagTask` (優先級 `osPriorityLow`)**：低優先級診斷任務。負責在地面調試階段透過序列埠輸出感測器自檢狀態，或接收終端指令。
* **`loraTxTask` (優先級 `osPriorityLow`)**：下行遙測傳送任務。以 10Hz~20Hz 頻率將當前的飛控狀態與感測器資料打包，並驅動 LoRa 無線模組送出。

### 1.2 中斷處理 (Interrupts) 與 NVIC 優先級設定
航電系統嚴禁在任務層級以阻塞式輪詢（Polling）方式讀取感測器，否則會極大地消耗 CPU 時間並引入不確定的抖動。因此，我們使用硬體定時器中斷來精確控制採樣時間：
* **TIM3 (3.2 kHz)**：每 $312.5\ \mu\text{s}$ 觸發一次，讀取高 G 加速度計 ADXL375。
* **TIM6 (1.6 kHz)**：每 $625\ \mu\text{s}$ 觸發一次，讀取 BMI088 加速度計。
* **TIM7 (2.0 kHz)**：每 $500\ \mu\text{s}$ 觸發一次，讀取 BMI088 陀螺儀。
* **USART6 (GPS)** 與 **USART2 (板間鏈路)**：配置 DMA 循環接收加上**空閒中斷 (IDLE)**，使得當資料流中斷（一幀結束）時，立即觸發中斷服務程式進行背景解析，完全避免任務主動查詢。

### 1.3 FreeRTOS 系統呼叫限制與中斷防護
在 Cortex-M 架構中，中斷優先級數值越小，物理優先級越高。
* 系統設定 `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY = 5`，這意味著優先級為 **0 到 4** 的中斷具有極高的即時性，不受 FreeRTOS 核心干預，但也**絕不能呼叫任何 FreeRTOS 的 API**（例如 `xQueueSendFromISR`），否則會導致核心狀態損毀。
* 優先級為 **5 到 15** 的中斷則可以安全地呼叫 `*FromISR` 結尾 of 的 API。
* 在本專案中：
  - 各採樣定時器（TIM3, TIM6, TIM7）的 NVIC 優先級被配置為 **6**，處於安全臨界區以下，既保證了極高的中斷響應速度，又保留了必要時與 RTOS 互動的能力（儘管驅動內部出於極致性能考量，刻意避免了任何核心呼叫）。
  - USB 中斷（OTG_FS_IRQn）配置為 **4**，由於其高於 syscall 限制，該中斷服務函式內部被嚴格禁止呼叫任何 RTOS 機制。

### 1.4 避免資料撕裂的 Ping-Pong 雙緩衝機制 (Double Buffering)
高頻中斷在背景寫入緩衝區，而 `defaultTask` 在前景讀取資料。若兩者同時存取同一個記憶體區塊，會導致正在讀取的 3D 加速度計讀到「前 2 軸是新資料，第 3 軸是舊資料」的資料撕裂（Data Tearing）現象。
為了解決這個問題，系統設計了 **Ping-Pong 雙緩衝機制**。以 ADXL375 為例：
```c
static volatile float g_adxl_batches[2][32][3]; // 雙緩衝二維陣列
static volatile uint8_t g_adxl_active_batch = 0; // 當前中斷寫入的緩衝區索引
static volatile uint16_t g_adxl_sample_cnt = 0;
```
* **中斷寫入端**：TIM3 中斷將資料寫入 `g_adxl_batches[g_adxl_active_batch][g_adxl_sample_cnt]`。當 `g_adxl_sample_cnt` 達到 32 筆時，中斷服務程式會執行：
  ```c
  g_adxl_active_batch = !g_adxl_active_batch; // 切換至另一個緩衝區
  g_adxl_sample_cnt = 0;
  g_adxl_ready_batch_idx = !g_adxl_active_batch; // 標記已填滿的緩衝區索引
  g_adxl_new_batch_ready = 1; // 設置就緒旗標
  ```
* **任務讀取端**：`defaultTask` 檢測到 `g_adxl_new_batch_ready == 1` 時，清除旗標，並直接讀取穩定且不再被中斷修改的 `g_adxl_batches[g_adxl_ready_batch_idx]` 資料進行降採樣，整個過程無鎖且無衝突。

### 1.5 CCM RAM 核心耦合記憶體優化
STM32F407 晶片除了常規的 112KB SRAM1/SRAM2 外，還具備一塊獨立的 **64KB CCM RAM (Core-Coupled Memory)**。
* **物理特性**：CCM RAM 直接連接至 Cortex-M4 的 D-bus（資料匯流排），與 CPU 核心同頻且具備零等待週期（Zero Wait-State）的高速存取特性。然而，它**不連接至 AHB 匯流排矩陣**，這意味著所有 DMA 週邊（如 SDIO、USART、SPI DMA）都**無法**讀取或寫入 CCM RAM。
* **優化配置**：我們將運算密集度極高、且不涉及週邊 DMA 傳輸的 **EKF 狀態變數 (`EKF_x`、`EKF_P`、`EKF_q`)**、**EKF 任務控制塊 (`StaticTask_t EKF_TaskControlBlock`)** 及 **`EKF_TaskStack` 堆疊空間**，透過 `CCMRAM` 巨集強行指定編譯至 `.ccmram` 記憶體區段：
  ```c
  #define CCMRAM __attribute__((section(".ccmram")))
  static float EKF_x[6] CCMRAM;
  ```
  這樣做既榨乾了 CPU 的計算潛能，又將常規 SRAM 釋放給了需要頻繁 DMA 寫入的 SD 卡（FatFs）和序列埠通訊，實現了最佳系統資源配置。

## 2. C/C++ 語法核心與暫存器操作 (C/C++ Language Core)

在深入分析原始碼之前，我們先複習嵌入式 C 語言中最核心的語法機制。

### 2.1 指標 (Pointers) 與記憶體位址
指標是存放記憶體位址的變數。在驅動中，我們常用指標來傳遞複雜的資料結構或控制硬體週邊。例如：
```c
HAL_StatusTypeDef ADXL375_Init(SPI_HandleTypeDef *hspi);
```
這裡 `hspi` 是一個指向 `SPI_HandleTypeDef` 結構體的指標。在函式內部，我們使用 `->` 運算子（成員存取指標運算子）來存取該結構體的成員：
```c
hspi->Instance->DR = val; // 往 SPI 資料暫存器寫入值
```
使用指標傳遞結構體的好處是**避免了記憶體複製的開銷**（只需傳遞一個 32 位元的位址），且能直接修改指標指向的實際資料。

### 2.2 結構體 (Structs) 與記憶體對齊 (`__attribute__((packed))`)
預設情況下，32 位元編譯器會為了對齊效率而在結構體成員之間插入填充空間（Padding）。例如：
```c
struct StandardStruct {
    uint8_t a;  // 1 byte
    uint32_t b; // 4 bytes (前面會被補 3 bytes 使得位址是 4 的倍數)
}; // 佔用 8 bytes
```
然而，在設計**下行遙測封包**或**板間通訊協定**時，我們的二進位資料必須是緊湊的，容不得任何空白填充。為此，我們使用 `__attribute__((packed))` 屬性告訴編譯器「取消對齊填充」：
```c
typedef struct __attribute__((packed)) {
    uint8_t  sync0;   // Offset 0
    uint8_t  sync1;   // Offset 1
    uint8_t  seq;     // Offset 2
    uint8_t  state;   // Offset 3
    uint32_t tick_ms; // Offset 4
} LinkPacket_t; // 緊湊排列，不含任何填充位元組
```

### 2.3 暫存器按位操作：位移 (`<<`)、按位與 (`&`) 與按位或 (`|`)
在硬體驅動中，我們需要頻繁地讀取和修改感測器的暫存器。
1. **拼接暫存器高低字節**：感測器的 16 位元數據通常拆成兩個 8 位元暫存器（如 `data_L` 和 `data_H`）。拼接方法如下：
   ```c
   int16_t raw_val = (int16_t)((rx_buf[1] << 8) | rx_buf[0]);
   ```
   * `rx_buf[1] << 8`：將高位位元組向左移動 8 位元，移到高 8 位。
   * `| rx_buf[0]`：透過按位或（OR）運算，將低位位元組填入低 8 位，拼成一個 16 位元整數。
2. **清除與置位特定位元**：
   * **置位 (Set bit)**：`reg_val |= (1 << 3);` —— 將暫存器的第 3 位置為 1，其餘位元保持不變。
   * **清除 (Clear bit)**：`reg_val &= ~(1 << 3);` —— 將暫存器的第 3 位清除為 0，其餘位元保持不變。

## 3. 飛控底層與核心原始碼模組解析 (Custom Code Overview)

以下是全專案 12 個核心模組的結構解說。讀者可以雙擊並打開 **`教學_6_19.html`**，切換到 **「🔍 原始碼逐行解析」** 標籤，然後使用下拉選單選擇對應的 `.c` 或 `.h` 檔案，即可在網頁中查看每一行程式碼的詳細說明與語法解鎖：

1.  **I2C 軟體總線 (`soft_i2c.h` / `soft_i2c.c`)**：利用普通 GPIO 模擬 I2C 時序讀寫 MMC5983MA。
2.  **高 G 加速度計驅動 (`adxl375.h` / `adxl375.c`)**：設定 ADXL375 的 3.2kHz SPI 數據讀取與 $49\text{mg}$ 解析度運算。
3.  **高性能 IMU 驅動 (`bmi088.h` / `bmi088.c`)**：讀取高精度的低 G 加速度與陀螺儀角速度。
4.  **氣壓高度計驅動 (`bmp388.h` / `bmp388.c`)**：調用官方 Bosch BMP3 API 進行溫度與氣壓浮點補償解算。
5.  **地磁計與校準 (`mmc5983.h` / `mmc5983.c`)**：包含 SET/RESET 固定脈衝，消去環境干擾與硬體磁滯。
6.  **GPS 解析核心 (`gps.h` / `gps_parse.h` / `gps.c`)**：USART6 DMA 循環加空閒中斷背景接收，實現 NMEA 語句校驗與經緯度實時換算。
7.  **姿態高度融合 EKF (`ekf.h` / `ekf_guard.h` / `attitude_math.h` / `ekf.c`)**：高優先級 1kHz EKF 導航演算法，運行於 CCM RAM，利用氣壓高度和重力校正四元數姿態。
8.  **有限狀態機 (`fsm.h` / `fsm.c`)**：狀態機更新邏輯，判斷發射、滑行、頂點、下落並控制雙踢火火工品。
9.  **下行遙測打包 (`telemetry.h` / `telemetry.c`)**：利用緊湊結構體包裝飛行狀態與感測器數據。
10. **多板冗餘備用鏈路 (`link.h` / `link_hw.h` / `link_proto.h` / `link.c` / `link_hw.c` / `link_proto.c`)**：USART2 板間通信與備用點火仲裁（BackupGate）。
11. **Flash 環形儲存 (`w25qxx.h` / `flash_ring_math.h` / `w25qxx.c`)**：W25Q256 的 SPI 物理讀寫與日誌環形指針管理。
12. **系統入口與主調度 (`main.h` / `main.c`)**：FreeRTOS 核心調度起點、TIM 中斷 Ping-Pong 緩衝配置與看門狗餵狗。

## 4. 之後如何修改代碼 (Code Modification Guide)

### 4.1 如何新增感測器
1.  **引腳配置**：在 STM32CubeMX 中設定 CS 或 I2C 引腳，生成代碼。
2.  **驅動編寫**：參考 `adxl375.c`，編寫感測器初始化與物理值讀取。
3.  **高頻中斷採樣**：將讀取動作加入定時器中斷（如 TIM6），寫入 Ping-Pong 寫入端。
4.  **前景對齊讀取**：在 `main.c` 的主任務中，檢查就緒標誌，讀取降採樣資料並封裝。

### 4.2 修改開傘參數
1.  打開 `Main_Code/Core/Inc/fsm.h`。
2.  修改起飛加速度巨集 `LIFTOFF_ACCEL_THRESHOLD` 或主傘高度巨集 `TARGET_MAIN_ALTITUDE`。
3.  在 `fsm.c` 的 `FSM_Step()` 中調整轉移規則。

## 5. 編譯、Make 與開發環境 (Toolchain & Build Configurations)

### 5.1 編譯四步驟
1.  **預處理**：處理巨集替換與標頭檔展開，生成 `.i`。
2.  **編譯**：將 C 代碼優化並翻譯成組合語言，生成 `.s`。
3.  **彙編**：將組合語言翻譯成二進位機器碼，生成物件檔案 `.o`。
4.  **連結**：根據 `.ld` 連結腳本，分配變數與程式碼實體位址，輸出 `.elf` 與 `.bin`。

### 5.2 Make 與 Makefile
Make 工具藉由讀取 Makefile 規則，實現增量編譯（只編譯修改過的文件），大幅提高開發效率。

### 5.3 CubeIDE 路徑配置
手動新增原始碼後，必須在屬性中配置 C/C++ General -> Paths and Symbols，將標頭檔目錄加入 Include Paths，將 `.c` 目錄加入 Source Location，否則會出現找不到檔案的編譯或連結錯誤。
