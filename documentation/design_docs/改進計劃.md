# RocketCom 改進計畫(飛行安全優先)

> 產出日期:2026-06-11(`total_test` branch)。本文件為整體程式碼審計後制定的分級改進路線圖。
> 優先原則:**火箭尚未首飛**,P0 = 任何會導致「該開傘時不開傘 / 不該點火時點火 / 失機」的問題,飛行前必須全數完成;程式碼整潔度與工具鏈問題往後排。
> 驗證雙軌:每項安全變更必附 **host 單元測試**(`cd tests && make run`,比照 `test_sensor_axis.c` 模式)+ **HIL 上板驗證**(test_runner.py,PD13 接 LED、無火工品)。

## 一、風險總覽表(審計發現,已逐項對源碼驗證)

  └→ P0-B 頂點失效保護 + baro 趨勢交叉檢查

### P0-B:頂點失效保護 + baro 原始趨勢交叉檢查(修 R1)

| 項目 | 內容 |
|------|------|
| 變更檔案 | `fsm.c/h`、`main.h`(參數)、`telemetry.c/h`(flag)、`tests/test_fsm.c` |
| 絕對計時器 | `FSM_FAILSAFE_APOGEE_MS 15000U`:起飛起算,**BOOST 與 COAST 都生效**(burnout 判定失效卡 BOOST 也救),超時強制轉 APOGEE 點火,遙測標 `TELEM_FLAG_FAILSAFE 0x40` |
| baro 交叉檢查 | COAST 中追蹤 baro 原始相對高度滾動峰值,`峰值 − 當前 ≥ 10.0m` 連續 20 週期(200ms)即成立 apogee(第三 OR 條件,完全不依賴 EKF;同時是 EKF unhealthy 時的主開傘路徑) |
| 起飛冗餘 | 起飛判定加第三條件 `baro_alt_rel > 20.0m`;pad_ref 於 PAD 期每 30s 緩慢重零(抗氣象漂移),起飛後凍結 |
| 參數理由 | 15s:由 150m 主傘目標 + 25s 全程看門狗反推,此級別火箭頂點 ~8–12s,15s 高於合理頂點 ≥3s 且比主傘看門狗早 10s。10m:≈BMP388 噪聲(σ≈0.6m)的 16σ,不致誤觸;頂點後自由落體 10m 僅 ~1.4s |
| Host 測試 | EKF 卡升+baro 正常 → 交叉檢查於真頂點 ±1.5s 點火;EKF+baro 全死 → 計時器 15.00s 點火恰一次;baro 死 → 原 EKF 路徑不變;卡 BOOST → 計時器仍點火 |
| ⚠️ 飛行前義務 | **`FSM_FAILSAFE_APOGEE_MS` 為無模擬資料下的保守暫定值,飛行前必以 OpenRocket 重推 `1.5 × t_apogee_sim`(且 ≥ t_apogee_sim+5s、≤ 主傘看門狗 −5s)** |

### P0-C:EKF 防護最小集(修 R3)

| 防護 | 參數 | 理由 |
|------|------|------|
| baro 創新值閘控 | 拒收 `\|y\| > max(5·sqrt(S), 25.0m)` | 5σ 標準卡方閘;25m 絕對下限防收斂後(5σ≈3m)誤拒真實氣壓暫態 |
| 持續拒收偵測 | 連續 ≥200 次(@200Hz≈1s)→ `EKF_HB_BARO_DIVERGE` | 區分單發雜訊與估計已發散;不吞壞值,由 FSM 降級接手 |
| P 對角夾限 | 位置 ≤1e4 m²、速度 ≤2.5e3 m²/s²、下限 1e-6 | 防 baro 中斷時 P 無界增長、恢復瞬間增益爆炸 |
| 狀態理智界限 | pos_z∈[−100,+2000]m、vel_z∈[−200,+350]m/s、水平 \|pos\|≤5000m、\|vel\|≤200m/s | 由本箭剖面(150m 主傘、25s、數百米頂點)放大一個數量級 |
| NaN 掃描 | 每 buffer 一次;NaN → P 重設、vel 清零、pos_z 拍至最近接受的 baro、q 正規化或 identity;`EKF_HB_NAN` 黏滯 5s | NaN 即全毀,必須重建 |
| baro 逾時 | >500ms 無被接受的 baro → `EKF_HB_BARO_TIMEOUT`;FSM 端另查 EKF 最後更新距今 >300ms 視同 unhealthy | 涵蓋 EKF_Task 餓死/queue 斷流 |

- 變更檔案:新增 `Core/Inc/ekf_guard.h`(純 header-only)、`tests/test_ekf_guard.c`;修改 `ekf.c/h`(掛載點:UpdateBaroDelayed 入口、predict 收尾、buffer 迴圈尾)、`telemetry.c/h`(`TELEM_FLAG_EKF_UNHEALTHY 0x10`)
- API:`EKF_GetHealthBits()` / `EKF_IsHealthy()`(critical-section 快照,同 EKF_GetState 模式)
- **FSM 降級策略**(EKF unhealthy 時,fsm.c 內 raw-baro 鏈):頂點 = P0-B 趨勢規則;主傘 = `h_fb ≤ 200m` 或 25s 看門狗;落地 = 2s 內 |Δh_fb|<2m 且 h_fb<30m
- HIL:浸泡期健康位恆 0(誤報率,**誤報降級本身也是風險**);`FAULT_INJECT_BARO_JUMP` 旗標驗拒收

### P0-D:感測器健康最小集(修 R4)

- 新增 `Core/Inc/sensor_health.h`(純邏輯)+ `tests/test_sensor_health.c`;main.c 批次提取點掛 Feed
- 偵測:**失流**(批次間隔 >100ms,正常 10ms)、**卡死**(raw 連續同值 ≥1s;ADC 噪聲使正常感測器幾乎不可能)、**範圍**(baro 26–110kPa、temp −40~85°C、|highg|≤250g,越界持續 0.5s)
- **消費端(本項重點,偵測而不消費沒有意義)**:
  1. baro fault → EKF 樣本 `has_baro=0` → 觸發 `EKF_HB_BARO_TIMEOUT` → FSM 降級鏈
  2. FSM 輸入快照帶 `sensor_bits`(起飛冗餘、fallback 閘控)
  3. telemetry `TELEM_FLAG_SENSOR_FAULT 0x20` + 1Hz `[HEALTH]` 診斷行
- 失效鏈(每環有 host 測試):IMU 卡死 → EKF 漂移 → baro 創新值持續拒收 → DIVERGE → FSM 降級 raw-baro → 仍可開傘

### P0-E:飛行迴圈阻塞消除(修 R6、R8)

1. **飛行中擦除禁令**:FSM 態 ∈ BOOST..MAIN_DEPLOY 不做滾動擦除;下一 sector 未預擦 → 丟該筆 + `g_ring_drop_count++`(遙測可觀測)
2. **PAD 期背景預擦**:目標 64 sectors(64×51÷50Hz ≈ 65s 飛行量,>2× 全程);開機仍擦 10 個維持開機時間,其餘 PAD/INIT 態每 1s 擦 1 個(單次最壞 400ms < IWDG 2.05s)。**發射檢核表:上架後等待 `[FLASH] pool=64` 再起飛** |
3. **ADC**:`ADC_Read_Battery_mv` 逾時 10ms→2ms;呼叫頻率降至 1Hz(`g_bat_voltage_mv` 本就是快取)
4. **printf 閘控**:BOOST..MAIN_DEPLOY 抑制 10Hz CSV 遙測行(每次 ~8.7ms;SD/Flash/LoRa 記錄不受影響);FSM 事件單行保留(已在動作之後)
5. **可觀測性**:每秒印 `[LOOP] max_ms` 單迴圈最大耗時水位;HIL 斷言飛行態 <20ms、`g_ekf_queue_drops` 不再遞增

### P0-F:熱啟動安全化(修 R7)

- main.c 熱啟動塊(1905-1921)重寫為 `HotStart_TryRestore()`,**驗證鏈全過才恢復,任一失敗回 STATE_PAD 完整重校準**:
  1. `FlashRing_GetLastPacket` CRC 有效
  2. SysFlags 校準區 magic+CRC 有效才設 `EKF_calibrated=1`
  3. 合理性:`pkt.tick_ms < 60000`(飛行不可能超 60s)、封包 baro 與當下首筆 baro 差 <300m
  4. **重點火政策:`pkt.flags & 0x01`(drogue 已點火)→ 恢復至 STATE_DESCENT,杜絕二次點火**;未點火且原態 COAST → 恢復 COAST 可正常點火
- 成功置 `TELEM_FLAG_HOTSTART 0x80`;熱啟動路徑跳過開機預擦迴圈(省 ~2s)
- Host 測試:四象限(已/未點火 × 各態)+ 驗證鏈各單項失敗回 PAD;HIL:bench 飛行中斷電重啟,斷言恢復態正確、PD13 不重複導通(請使用者執行)
