# 🚀 成大 ISP 航電主控系統 — 待確認與改進清單 (To-Confirm & Improvement Checklist)

此清單用於追蹤系統在實際上架、模擬發射與實彈測試前，必須進行硬體對齊、軟體數值推導或系統壓測的待確認事項。

---

## 📋 待確認清單 (To-Confirm)

- [ ] **1. 開傘機構與持續時間確認**
  - **副傘點火導通時間：** 目前在 `fsm.h` 中 `FSM_PYRO_HOLD_MS` 設定為 2000ms (2秒)。需配合實際點火頭與火藥量，測試 2 秒的導通時間是否能保證點燃，且不會因高溫熔毀短路引起板子死機。
  - **主傘釋放舵機角度：** 目前主傘釋放舵機在 `STATE_MAIN_DEPLOY` 下以 TIM4_CH3 PWM (CCR=2000) 控制轉動。需現場實測舵機的機械連桿與扣具角度，確保鎖定 (0°) 與釋放 (90°) 位置不會卡死。

- [ ] **2. 頂點絕對失效保護安全時間鎖推導**
  - **背景：** `FSM_FAILSAFE_APOGEE_MS` 目前暫定為 15000ms (15秒)。
  - **待辦：** 必須在發射前以 **OpenRocket** 進行該級火箭的彈道模擬，取 $1.5 \times t_{\text{apogee\_sim}}$ (模擬頂點時間) 重新填入。必須確保此值大於模擬頂點時間 5 秒以上，且小於主傘看門狗時間 5 秒以上。

- [ ] **3. 主傘安全看門狗時間確認**
  - **背景：** `FSM_MAIN_WATCHDOG_MS` 目前設定為 25000ms (25秒)。
  - **待辦：** 需結合發射場高度與降落速度模擬，確認 25 秒是否足夠讓火箭完全度過頂點下墜至安全開傘高度，避免在高速下墜中強開主傘導致傘繩扯斷。

- [ ] **4. ADXL375 高-G 加速度計軸向對齊實機驗證**
  - **背景：** 因 PCB 佈線軸向相反，`sensor_axis.h` 中對 ADXL375 的 X 與 Y 軸數據進行了取負值 (Inversion) 的修正。
  - **待辦：** 在地面搖晃或震動航電板，在 GUI Monitor 上對比 `BMI088` 與 `ADXL375` 的三軸輸出，確保在同一 Body Frame (X=右, Y=前, Z=上) 下，重力方向與震動方向方向完全一致。

- [ ] **5. SD 卡插拔檢測與無卡防空狀態驗證**
  - **背景：** `SDIO_DET` 引腳 (PD3) 已從 `GPIO_NOPULL` 改為內建 `GPIO_PULLUP` 以防止無卡時總線超時卡死主迴圈。
  - **待辦：** 測試在「拔除 SD 卡」的狀態下開機，確認系統能優雅跳過掛載日誌流程，且 LoRa 遙測與 EKF 任務能 100% 正常工作、無當機復位現象。

- [ ] **6. 地磁計 MMC5983MA 的 SET/RESET 橋偏校準頻率**
  - **背景：** 目前在非飛行狀態下每 10 秒執行一次 `MMC5983_Recalibrate()` 來補償溫飄。
  - **待辦：** 確認在發射架上等待發射時，地磁融合角度是否會因為校準產生跳變；飛行中地磁計已被隔離，但需確認這 10 秒一次的 recalibrate 不會影響起飛判定。

## 🛠️ 改進與優化清單 (Improvements)

- [ ] **1. Flash Ring Buffer 寫入頻率與 Sector 擦除分離 (極重要)**
  - **現狀：** 目前飛行階段寫入頻率限制在 50Hz 而非 100Hz，因為 `FlashRing_WritePacket` 內含 Sector 擦除（最壞需耗時 400ms 阻塞），若開到 100Hz 會卡住主任務的數據同步。
  - **目標：** 將 Flash 擦除邏輯移到獨立的低優先級 FreeRTOS 任務中，透過 `g_spi3_mutex` 互斥鎖與 `StartDefaultTask` 協調，將飛行寫入頻率安全提升至 100Hz。

- [ ] **2. 壓測 SPI3 總線並發性能 (SX1268 LoRa + W25Q128 Flash)**
  - **現狀：** Flash 寫入與 LoRa 遙測共同掛載在 SPI3 總線上。
  - **目標：** 進行壓力測試，模擬同時進行 50Hz 數據寫入與 5Hz 遙測傳輸，確認互斥鎖沒有造成優先級反轉 (Priority Inversion) 或導致高優先級飛控任務的延遲。

- [ ] **3. SPI3 共用匯流排單點故障隔離與超時保護 (避免單一故障拖垮總線)**
  - **現狀：** Flash 寫入（W25Q128）與 920MHz LoRa 模組（E80）共用 `SPI3` 匯流排。若單一設備損壞（如 MISO 接地短路或晶片內部故障），一旦 SPI 傳輸函數或忙碌等待（如 `W25QXX_WaitForReady`）在無防護的 `while` 迴圈中卡死，或因佔用 `g_spi3_mutex` 互斥鎖不放，將導致整個 SPI 總線癱瘓，波及其他共用設備並造成主控程序阻塞。
  - **目標：** 
    1. 在所有 SPI 傳輸與等待 Ready 的等待迴圈中加入嚴格的計數器超時機制，禁止無限 `while` 等待。
    2. 建立硬體健康評估機制。若開機 ID 讀取異常（如讀到全 `0x00` 或全 `0xFF`）或執行中 SPI 連續傳輸超時失敗達到設定閥值，應自動標記該設備為「失效（offline）」，並在後續程序中主動跳過對該設備的 SPI 存取與 `g_spi3_mutex` 鎖定，確保其他正常設備能繼續穩定運作。
