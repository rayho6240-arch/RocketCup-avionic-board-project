## 二、 已進行的驗證記錄歷程 (Verification Records)

以下記錄專案開發過程中，各項需求與功能的實體/模擬驗證歷程：

### 1. [已驗證] 電子羅盤實體線路接反與 GPS 鮑率錯誤戰術救援
* **驗證時間**：2026-06-03
* **對應需求**：REQ-SYS-05 (導航定位相關硬體)
* **驗證方式**：
  * **硬體現況**：發現 PCB 走線中 I2C1 SCL 與 SDA 接腳與晶片硬體定義相反，且 GPS 預設為 38400 bps 與原代碼 115200 bps 不符。
  * **驗證手段**：實作軟體模擬 I2C 驅動（SCL=PB7, SDA=PB8，採開漏輸出），並將 GPS 串口 (USART6) 鲍率修正為 38400 bps。透過 USART2 串口遙測觀測數據。
  * **驗證結果**：地磁計 MMC5983MA 成功讀取 (`ok=921, err=0`)，GPS NMEA 數據流零校驗錯誤流暢解析 (`ok=60, err=0`)，採樣率監測器顯示正常。

### 2. [已驗證] SD 卡高頻數據全程日誌寫入與無卡防卡死優化
* **驗證時間**：2026-06-01
* **對應需求**：REQ-SYS-05 (導航定位與數據保存)
* **驗證方式**：
  * **驗證手段**：模擬飛行過程進行 SDIO 寫入。配置 `SDIO_DET_Pin` 為上拉電阻，以防引腳浮空卡死；並在 `StartDefaultTask` 內實作 FSM 狀態驅動：升空至落地前以 100Hz 記錄，落地後降為 10Hz，並於落地超過 30 分鐘或按壓 `USER_BT1` 時安全關閉檔案。
  * **驗證結果**：SD 卡於無卡狀態下能優雅跳過，有卡時能自動以 `HIL_001.CSV`～`HIL_999.CSV` 遞增建檔，高頻連續寫入無卡死，每秒 `f_sync` 寫入成功。

### 3. [已驗證] Buzzer 開機提示音與 32-bit ARR 溢出修正
* **驗證時間**：2026-06-01
* **對應需求**：系統狀態反饋
* **驗證方式**：
  * **驗證手段**：配置 TIM2 CH1 PWM，控制蜂鳴器在初始化結束後發出 2kHz -> 4kHz 的兩聲漸高音。針對 ARR 動態縮小時若 CNT 已大於新 ARR 會導致等候 4295 秒的 Bug，改用手動觸發 `EGR = TIM_EGR_UG` 即時歸零 CNT。
  * **驗證結果**：蜂鳴器發聲宏亮，且連續多次重啟皆能穩定發出開機兩聲提示音。

### 4. [已驗證] Flash 環形日誌緩衝區 (Ring Buffer) HIL 驗證
* **驗證時間**：2026-05-27
* **對應需求**：REQ-SYS-05 (飛行數據保存)
* **驗證方式**：
  * **驗證手段**：透過 SPI3 匯流排對 W25Q128 進行 30 秒硬體迴路測試 (HIL)。以 20Hz 寫入 52-byte packed 封包，並在寫入前執行二分搜尋（Binary Search）快速恢復寫入頭，同時進行 10 個 Sector (40KB) 預擦除。
  * **驗證結果**：30秒內寫入 461 筆封包，透過串口 Dump 讀回，所有 magic bytes (`0xAA 0x55`) 與 CRC-16/CCITT 校验 100% 通過。

### 5. [已驗證] 高頻定時器中斷與雙緩衝 (Ping-Pong) 感測器採樣
* **驗證時間**：2026-05-27
* **對應需求**：REQ-SYS-05 (高頻運動數據獲取)
* **驗證方式**：
  * **驗證手段**：設定 TIM6 (1.6 kHz 中斷讀 BMI088 加速度計)、TIM7 (2.0 kHz 中斷讀 BMI088 陀螺儀)、TIM3 (3.2 kHz 中斷讀 ADXL375 高 G 加速度計)。中斷內資料打包存入雙緩衝區，任務層每 10ms 讀取最新批次。
  * **驗證結果**：實測物理採樣頻率精準達到設計極限：BMI088 加速度 1600.0Hz、陀螺儀 2000.0Hz、ADXL375 3200.0Hz，且無總線衝突，CPU 上下文切換開銷降低 96.8%。

### 6. [已驗證] 獨立看門狗 (IWDG) 快速復位保護
* **驗證時間**：2026-05-27
* **對應需求**：系統容錯安全
* **驗證方式**：
  * **驗證手段**：配置 IWDG 使用 LSI 32kHz 振盪器，除以 16 分頻，載入值 4095，將超時設為 2.05 秒。在主任務主迴圈中每毫秒進行一次餵狗。
  * **驗證結果**：當任務正常運行時系統無誤復位；當人工在代碼中加入死迴圈阻塞時，系統在精準的 2.05 秒重啟。

## 三、 當前待驗證與尚未驗證項目 (Pending Verifications)

1. **空中斷電熱啟動恢復功能 (REQ-SYS-06)**
   * **狀態**：**尚未驗證** (代碼已完成，等待硬體 HIL 測試)。
   * **驗證方式與步驟**：
     * **[1] 斷電模擬**：在發射台 PAD 狀態完成 3 秒自檢校準（此時已將 biases 寫入 Sector 0）。模擬點火升空進入 `STATE_BOOST` 或 `STATE_COAST`，並在此期間手動切斷系統電源 2~3 秒後重新接通。
     * **[2] 狀態恢復觀測**：重新上電後，觀察 UART 終端機與遙測數據是否成功跳過 3 秒自檢校準，加載 `reboot_count` 並遞增，繼承斷電前的 FSM 狀態。
     * **[3] 數據平滑性**：驗證是否正確讀取 Sector 0 的基準高度與 biases，並利用 Ring Buffer 最後兩筆數據差分估算 Z 速度，將高度、垂直速度與四元數姿態平滑注入 EKF，無階躍發散。

2. **20s 內手動 EKF 重新校準與高度歸零 (USER_BT1 短按與 3kHz 雙短鳴)**
   * **狀態**：**尚未驗證** (代碼已完成，等待上板測試)。
   * **驗證方式與步驟**：
     * **[1] 觸發測試**：上電開機 20 秒內，傾斜航電板使姿態估計偏離。按下 `USER_BT1` 鍵。
     * **[2] 確認反饋**：確認蜂鳴器響起非阻塞的 3kHz 雙短鳴（確認音），且遙測終端輸出 `[EKF] Static calibration reset triggered!`。
     * **[3] 校準重建**：確認系統重新執行 3000 筆感測器數據的靜態偏差估計，校準完成後自動調用 `EKF_SaveCalibrationToFlash` 將最新的零偏寫入 Flash Sector 0。

3. **開機手動清除 Flash 狀態 (USER_BT2 擦除 Ring Buffer 標記區)**
   * **狀態**：**尚未驗證** (代碼已完成，等待上板測試)。
   * **驗證方式與步驟**：
     * **[1] 擦除測試**：上電開機時按住 `USER_BT2` 鍵。
     * **[2] 確認反饋**：確認蜂鳴器響起 300ms 的持續蜂鳴聲，且終端輸出 `[SYS] Flash Ring Buffer 標記已成功擦除！`。
     * **[3] 預期結果**：系統應完全重置 Ring Buffer 首尾 Sector 標記。此時即使前一次飛行中斷，開機重啟亦判定為正常地面起飛，進入 `STATE_PAD` 模式。

4. **80-Byte telemetry 封包與落海 1Hz 記錄速率變更**
   * **狀態**：**尚未驗證** (代碼已完成，等待上板測試)。
   * **驗證方式與步驟**：
     * **[1] telemetry 格式對齊**：觀察 Ring Buffer 寫入的封包格式，大小已調整為 80-byte。落地後導出 Flash 與 SD 卡的 CSV 比對，確認 EKF 四元數（q0, q1, q2, q3）、高度、速度、GPS 等欄位 100% 一致。
     * **[2] 降頻紀錄**：當飛行狀態機進入 `STATE_LANDED`（安全著陸）後，確認 Flash 寫入頻率成功由 50Hz 降為 1Hz，且無 IWDG 餵狗溢出或 FSM 卡死。

5. **著陸一次性寫入系統旗標與任務總結 (Sector 0/1 防飛行擦除安全鎖)**
   * **狀態**：**尚未驗證** (代碼已完成，等待上板測試)。
   * **驗證方式與步驟**：
     * **[1] 飛行安全鎖驗證**：在起飛至下降的整個飛行過程中，監控 W25Q128 的 SPI3 總線，確認除 Page Program 外無任何 Sector Erase 指令發送，防止 CPU 卡死阻塞。
     * **[2] 著陸寫入驗證**：進入 `STATE_LANDED` 瞬間（g_touchdown_tick 首次記錄），觀察 Sector 0 與 Sector 1 被擦除寫入。讀出 static 區，確認最高高度、最大速度、過載極值、狀態時間戳、發射點/落地點 GPS 與 reboot 次數均成功寫入。
