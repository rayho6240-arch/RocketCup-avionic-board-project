## 專案重大調整與複用策略

由於**成本預算考量**，我們對航電硬體進行了極大化的複用與簡化，目前調整如下：

### 1. 首批送印板子 (共 5 張)

首批打樣送印僅選擇以下 5 張核心電路板：

1. `newpower` (位於 `BigAvionic/power layer_new`) - 主航電電源層板
2. `lora` (位於 `BigAvionic/lora`) - 主航電無線遙測通訊層板
3. `mainmcu` (位於 `BigAvionic/Second_layer_MCU`) - 主航電核心主控層板
4. `top` (位於 `BigAvionic/top_layer`) - 主航電頂層接口板
5. `Deployment board` (位於 `Deployment board/`) - 開傘部署控制板

### 2. 複用與替代策略 (Cost Optimization)

- **副航電備份板 (SmallAvionic) -> 由 `mcu layer` 代替**
  - 取消原先獨立設計的 `SmallAvionic` 系列（如 Joyce、Ray、Tank、Tony 版）。
  - 直接採用主航電四層疊板中的 `mainmcu` (MCU Layer) 單板作為備份副航電板，實現硬體完全一致性與零額外打樣費。
- **地面接收站 (GroundStation) -> 由 `BigAvionic` 整體四層疊板代替**
  - 取消原本獨立的 `GroundStation` 地面站 PCB 打樣。
  - 由於 `BigAvionic` 整體四層疊板已配備完整的 STM32 主控、LoRa 射頻收發與 USB-TTL 界面，地面接收站將直接使用一套完整組裝的 `BigAvionic` 四層疊板代替。

### 3. 預算最佳化方案 - 二合一合併板

- `Production_MCU_TOP_2_in_1` (原名 Production_MCU_LORA_2_in_1)
  - 實際上為 `top layer` (頂層板) 與 `mcu` (主控板) 合成的二合一合併板。
  - 用於作為未來量產或高度整合時的低成本備選方案（減少一張板子的打樣費用）。

## 實地測試里程碑 (Testing Milestones)

- **2026-06-01：Buzzer 開機提示音實作（兩聲漸高）**
  - 實測結果：透過 TIM2 CH1 PWM（timer clock = 1 MHz，輸出腳 PA0）實現開機兩聲提示音（2 kHz → 4 kHz，各 100 ms，間隔 100 ms），於 `StartDefaultTask` 的 `Flash_DumpAll` 之後、SD 卡初始化之前播放，代表系統初始化完成。過程中發現並修正 TIM2 ARR overflow bug：TIM2 為 32-bit timer，動態縮小 ARR 時若當下 CNT 已大於新的 ARR，計數器會一路數到 `0xFFFFFFFF`（約 4295 秒）才歸零，導致 buzzer 在第三聲後完全不響。修正方法為設定 ARR/CCR1 後立即寫入 `EGR = TIM_EGR_UG` 強制 update event，使 CNT 立刻歸零並套用新週期。

- **2026-05-27：printf 重定向 USART2 (115200 baud) 實現即時串口波形監控**
  - 實測結果：在 `main.c` 新增 `_write()` syscall 覆寫，將標準 `printf` 輸出重定向至 USART2（連接 ST-LINK 虛擬串口）。主迴圈每 50ms (20 Hz) 透過 UART 輸出 BMI088 加速度、ADXL375 高 G 加速度、BMP388 氣溫/氣壓/高度等共 9 個數值欄位的 CSV 格式串流，搭配 Serial Plotter / Python 腳本即可即時繪製 IMU 波形。每 1000ms 另輸出一次各感測器實際採樣率（`[RATE]` 格式），便於自動化測試斷言分析。

- **2026-05-27：採樣率監測模組 (Rate Monitor) 整合，全感測器實測 Hz 可視化**
  - 實測結果：新增獨立模組 `rate_monitor.h`，以巨集 `SAMPLING_RATE_DECL()` 宣告全域結構體 `g_sampling_rate`，包含 BMI088 加速度計、陀螺儀、ADXL375、BMP388 各通道的 `.rate_hz` 欄位。各中斷與任務分別呼叫 `RATE_TICK_BMI088_ACC()`、`RATE_TICK_BMI088_GYRO()`、`RATE_TICK_ADXL375()`、`RATE_TICK_BMP388()` 進行累加，模組每秒自動計算並更新真實物理採樣頻率。在 IDE Watch / Live Expressions 只需加入 `g_sampling_rate` 即可一次觀察全部感測器 Hz。可透過 `rate_monitor.h` 中 `#define RATE_MONITOR_ENABLE` 的有無開關整個功能，零停機代價。

- **2026-05-27：IWDG 獨立看門狗配置完成，更新為 2.0 秒快速超時以保障飛行安全**
  - 實測結果：配置 IWDG 使用 LSI 32 kHz 振盪器，16 分頻（計數頻率 2000 Hz），載入值 4095，超時時間精準設定為 **2.05 秒**。超短的超時時間可保障在火箭飛行中如遇強干擾死機時能以最快速度重啟。通過 FreeRTOS 將 SD 卡 / FatFs 磁碟讀寫等耗時任務完全隔離在 `DataLogging_Task`，主迴圈每個 `tick`（1 ms）均呼叫 `HAL_IWDG_Refresh()` 餵狗，保障系統高穩定性不誤復位。

- **2026-05-27：W25Q128 SPI Flash 啟動自檢與寫入校驗 100% 成功**
  - 實測結果：透過 SPI3 匯流排 (SCK=PB3, MISO=PB4, MOSI=PB5) 與 CS=PA15 成功讀取 W25Q128 JEDEC ID (`0xEF4018`)，完成 4KB Sector 0 擦除、61 字節頁寫入、逐字節讀回比對全數通過。驅動程式模組化設計，包含 `w25q128.h` / `w25q128.c`，實作寫使能、忙碌超時輪詢、頁寫入、連續讀取等完整 HAL SPI 阻塞傳輸 API。由於 SPI3 為獨立匯流排，與 SPI1 (ADXL375/BMP388) 和 SPI2 (BMI088) 完全無匯流排衝突。

- **2026-05-27：SD 卡 10 秒高頻數據日誌 (HIL_10S.CSV) 寫入與無卡插拔防卡死優化成功**
  - 實測結果：實現了在 MCU 啟動後自動延遲掛載 SD 卡，並進行為期 10 秒的 100 Hz CSV 高頻遙測日誌寫入（儲存至 `HIL_10S.CSV` 檔案中，包含系統 Tick、BMI088、ADXL375、BMP388 所有感測器數據）。在寫入 10 秒後安全關閉檔案並卸載磁碟。針對無實體 SD 卡插入的 HIL 測試場景，將 `SDIO_DET_Pin` (PD3) 引腳配置由 `GPIO_NOPULL` 優化為 `GPIO_PULLUP`，徹底避免了引腳浮空導致 MCU 假性插卡檢測，使系統無卡時能優雅跳過掛載與 `f_open` 寫入，保證串口遙測輸出的絕對流暢與穩定，零卡死！

- **2026-05-27：TIM6 + TIM7 定時器獨立中斷 + 雙緩衝區 (Ping-Pong Buffer) BMI088 加速度 1600 Hz & 陀螺儀 2000 Hz 測試成功**
  - 實測結果：配置硬體定時器 TIM6 觸發 1.6 kHz 中斷（只讀取 BMI088 加速度計並塞入 16 筆的 Ping-Pong 緩衝區），配置 TIM7 觸發 2.0 kHz 中斷（只讀取 BMI088 陀螺儀並塞入 20 筆的 Ping-Pong 緩衝區）。FreeRTOS 任務每 10ms (100Hz) 提取各自批次最新數據進行遙測。因為 SPI2 匯流排專用於 BMI088，無任何中斷-任務競爭衝突，完全免除軟體互斥鎖。實測加速度計與陀螺儀物理採樣頻率精準達到官方設計之最高物理上限：**加速度計 1600.0 Hz & 陀螺儀 2000.0 Hz**！

- **2026-05-27：TIM3 定時器中斷 + 雙緩衝區 (Ping-Pong Buffer) ADXL375 3200 Hz 測試成功**
  - 實測結果：利用硬體定時器 TIM3 觸發 3.2 kHz 中斷，在中斷內直接透過 SPI1 讀取 ADXL375 原始資料（耗時僅 5.6us），並存入 32 筆資料的 Ping-Pong 雙緩衝區。FreeRTOS 任務每 10ms (100Hz) 提取批次最新數據進行遙測，上下文切換開銷降低 96.8%。實測物理採樣頻率精準達到理論最高 **3200.0 Hz**！
  - 另外開啟 BMP388 至 200 Hz ODR，並加入 SPI1 總線互斥安全鎖，在讀取 BMP388 期間屏蔽 TIM3 中斷防止衝突。將系統 LED 翻轉減緩為溫和的 0.5 秒 (1 Hz) 心跳指示。

- **2026-05-25：頂層接口板 (Top Layer) -> Arduino 整合測試成功**
  - 測試結果：成功透過 Arduino (ESP32) 讀取 I2C 磁力計 (`MMC5983MA`) 與解析 UART2 GPS (`NEO-M9N`) 的實時同步數據流。所有實體走線與通信介面工作完美。
  - 目前狀態：等待首批 `BigAvionic` 板子打樣完成後，正式與 STM32 主控板進行韌體整合測試。

## 開發環境設定

### 1. 快速開始 (本地 IDE 輕量化 GCC 模式)

**必要工具鏈（已安裝）**：

- `arm-none-eabi-gcc` v16.1.0（透過 Homebrew 安裝於 `/opt/homebrew/bin/`）
- `arm-none-eabi-binutils` v2.46.0

**注意**：主晶片因缺料改用 **STM32F407VGT6**（Pinout 與 STM32F405VGT6 完全一致），氣壓計改用 **BMP388**（Pinout 與 BMP390 完全一致）。

### 2. 編譯步驟

1. 確認工具鏈在 PATH 中：
   ```
   arm-none-eabi-gcc --version
   ```
2. 執行編譯（使用 4 核心並行編譯）：
   ```
   make -j4
   ```
3. 編譯成功後，會在 `Main_AV_F407/Debug/` 目錄下自動產生以下執行與燒錄檔：
   - `bigav_407.elf` (用於除錯)
   - `bigav_407.hex` / `bigav_407.bin` (用於燒錄)

### 3. 一鍵燒錄與實體單步除錯

推薦在 IDE 中安裝 **Cortex-Debug** 擴充套件，並在專案根目錄 `.vscode/launch.json` 中配置 ST-LINK 除錯器，即可在 IDE 中按下 F5 實作 ST-Link 一鍵燒錄與中斷點除錯。

## 重要修訂紀錄

- **2026-06-01（改善項目實作 #1）**：依改善計畫（`context-rosy-kahan.md`）實作三項低風險項目，皆通過 STM32CubeIDE 工具鏈編譯。
  - **D — FPU context save 確認（無需修改）**：FreeRTOS 採 `ARM_CM4F` port + 編譯旗標 `-mfloat-abi=hard`（`__FPU_USED=1`，`SystemInit` 已設 CPACR）。`port.c` 於 `xPortStartScheduler` 呼叫 `vPortEnableVFP()` 並設定 `FPCCR` 的 `ASPEN|LSPEN`（硬體 lazy FPU context 保存），PendSV 依 EXC_RETURN 自動保存 S0–S31/FPSCR。`FreeRTOSConfig.h` 的 `configENABLE_FPU 0` 僅 ARMv8-M（CM23/CM33）port 會讀取，於 M4F 無作用——故 EKF 浮點運算在任務切換時本就安全。
  - **I — EKF queue drop 計數（觀測先行）**：`osMessageQueuePut` 失敗時遞增 `g_ekf_queue_drops`，於 `[RATE]` 行新增 `EKF_DROP` 欄位輸出。釐清一個計畫原稿的隱患：EKF queue 傳遞的是 **2 槽 ping-pong buffer 指標**（`g_ekf_buffers[2]`，索引以 `!idx` 切換），若僅把 queue 深度擴大到 8 而不同步擴大實體 buffer pool，會造成 stale-pointer 覆寫；且實測 `bss` 已達 ~113 KB / 128 KB SRAM，擴大為 8 buffer（+~22 KB）會溢位。故先計數觀測，確認確實有 drop 再評估「擴 pool + 改 modulo-N 索引 + 移入 CCM」。
  - **N — Flash dump 改條件觸發**：開機 `Flash_DumpAll(&hspi3)` 改為僅在按住 `USER_BT1`（上拉輸入，按下 = LOW）時執行，平時開機省下整顆 Flash 輸出的 ~3–5 秒。
